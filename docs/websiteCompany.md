# Website (Frontend) — Компани / олон дэлгүүр — Phase 1 spec

Зорилго: Эцсийн хэрэглэгчийн **вэб дэлгүүр** дээр компани (vendor) хэрхэн харагдах, URL, жагсаалт, барааны шүүлтийг тодорхойлох. Backend-ийн `Company` + `product.companyId` (`docs/addCompany.md`) дээр суурилна.

---

## 1. Бүтээгдэхүүний чиглэл (сонголт — Phase 1-д нэгийг сонгоно)

| Загвар | Тайлбар |
|--------|---------|
| **A. Нэг вэб, олон компани** | Нүүр хуудас дээр бүх компанийн бараа холимог; компаниар шүүлт + компанийн хуудас. |
| **B. Компани тус бүр subdomain** | `company-a.snu.mn` — DNS + frontend routing нэмэгдэл их. |
| **C. Зөвхөн нэг платформ брэнд** | Компани нь "дотоод этгээд", хэрэглэгчид зөвхөн бараа харагдана — компани UI бага. |

**Phase 1 санал:** **A** — хэрэгжилт хамгийн бага, SEO-д `/companies/[slug]` ашиглахад тохиромжтой.

---

## 2. Маршрут (routing) санал

| URL | Агуулга |
|-----|---------|
| `/companies` | Бүх **идэвхтэй** компанийн жагсаалт (лого, нэр, богино тайлбар — Phase 2 талбар нэмж болно). |
| `/companies/[slug]` | Тухайн компанийн танилцуулга + тухайн компанийн барааны жагсаалт (pagination, ангилал шүүлт). |
| `/shop` эсвэл одоогийн каталог | Олон компанийн нэгдмэл бараа: **компани шүүлтүүр** (optional chip/filter). |
| `/product/[id]` эсвэл `[slug]` | Барааны дэлгэрэнгүй дээр **"Энэ барааг нийлүүлсэн"** — компанийн нэр + линк `/companies/[slug]`. |

---

## 3. UI/UX шаардлага

### 3.1. Компанийн жагсаалт хуудас

- Картууд: лого, нэр, `active` компаниуд л.
- Хайлт: компанийн нэрээр (client эсвэл server — Phase 2).

### 3.2. Компанийн дэлгэрэнгүй

- Дээд хэсэг: лого, нэр, холбоо барих (хэрэв нийтэд зөвшөөрөгдсөн мэдээлэл).
- Доод хэсэг: барааны grid — одоогийн product card компонентийг дахин ашиглах, зөвхөн `companyId` эсвэл `companySlug`-аар шүүсэн API дуудлага.

### 3.3. Нэгдмэл дэлгүүр

- Шүүлтүүр: "Бүх компани" | `Company A` | `Company B` …
- Сонгосон шүүлтүүр localStorage эсвэл URL query (`?company=slug`) — хуваалцах боломжтой линк.

---

## 4. API хамаарол (frontend хүлээлт)

Phase 2 backend:

- `GET /api/companies/public` — зөвхөн `status: active`, нийтэд харуулах талбарууд л (имэйлийг нуух боломжтой).
- `GET /api/companies/public/:slug` — компани + meta.
- `GET /api/products?companyId=` эсвэл `?companySlug=` — одоогийн `GET /product/all` зэргийг өргөтгөх эсвэл шинэ endpoint.

**Анхаар:** Одоогийн backend-д дээрх public endpoint алга; Phase 1-д зөвхөн гэрээтэй.

---

## 5. SEO ба аюулгүй байдал

- Компани хуудас: `title`, `meta description` (`name`, slug-аас).
- Нууц мэдээлэл (`email`, дотоод статус) public JSON-д орохгүй.

---

## 6. Хоосон төлөв

- Компани байхгүй: `/companies` дээр тайлбар + дэлгүүр рүү линк.
- Компани байгаа ч бараа байхгүй: "Удахгүй шинэ бараа нэмэгдэнэ" гэх мессеж.

---

## 7. AdminPanel-тай уялдаа

- Super Admin компанийг `suspended` болговол вэб дээр **жагсаалт + бараа хоёуланд** харагдахгүй (backend filter заавал; frontend зөвхөн нэмэлт).

---

## 8. Phase 1 acceptance

1. Spec-д сонгосон routing загвар (A) баримтжуулсан.
2. Хэрэглэгчийн урсгал: нүүр → компани жагсаалт → нэг компани → бараа → барааны дэлгэрэнгүй → буцаад компани руу текстээр тодорхой.
3. Шүүлтүүрийн байрлал (нүүр эсвэл shop) нэг удаа шийдэгдсэн байна.
